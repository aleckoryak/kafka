Apache Kafka® is an open-source distributed streaming system used for stream processing, real-time data pipelines, and data integration at scale.

This topic describes how Kafka has been optimized for efficiency because efficiency is key component of effective multi-tenant operations.

For example, a primary use case for Kafka is the handling web activity data, which is very high volume; each page view may produce dozens of message writes. Each of these published messages is read by one or or more consumers, so consumption should be as as efficient as possible.

In a multi-tenant system, a small bump in application usage can result in a bottleneck in the downstream infrastructure service.

To help with this issue, Kafka is designed to be very fast. It is more likely that an application will tip-over under load before the Kafka infrastructure does.

There are two common causes of inefficiency in a multi-tenant system:

- Excessive byte copying
- Too many small I/O operations

The next sections describe how Kafka avoids the inefficiencies.

## Avoids excessive byte copying

Excessive byte-copying results in inefficiency. At low message rates, excessive byte copying is not an issue, but under load, the impact is significant. To avoid this, Kafka employs a standardized binary message format that is shared by the producer, broker, and the consumer so that data chunks are transferred without modification.

In contrast to a typical system where the operating system and application are doing multiple reads and writes, before data is sent over the network, Kafka expects multiple consumers for a topic. Using the zero-copy optimization offered by modern Unix and Linux operating systems with the [sendfile system call](https://man7.org/linux/man-pages/man2/sendfile.2.html), data is copied into the pagecache exactly once and reused on each consumption instead of being stored in memory and copied out to user-space every time it is read. This enables messages to be consumed at a rate that approaches the limit of the network connection.

Using this combination of pagecache and `sendfile` means that on a Kafka cluster where the consumers are mostly caught up you will see no read activity on the disks as they are serving data entirely from cache.

## Batches I/O operations

Too many small I/O operations can occur both between the client and the server and in the server’s own persistent operations. To avoid this, the Kafka protocol is built around a “message set” abstraction that naturally groups messages together. This allows network requests to group messages together and amortize the overhead of the network roundtrip rather than sending a single message at a time. The server in turn appends chunks of messages to its log in one go, and the consumer fetches large linear chunks at a time.

This simple optimization increases speed by orders of magnitude. Batching leads to larger network packets, larger sequential disk operations, contiguous memory blocks, and more, all of which enables Kafka to turn a bursty stream of random message writes into linear writes that flow to consumers.

## End-to-end batch compression

In some cases the bottleneck is actually not CPU or disk, but network bandwidth. This is particularly true for a data pipeline that needs to send messages between data centers over a wide-area network. Of course, the user can always compress their messages, one at a time, without support from Kafka. However, this can lead to poor compression ratios because of repetition between messages of the same type. For example, field names in JSON or user agents in web logs. Efficient compression requires compressing batches of messages together rather than compressing each message individually.

Kafka supports the compression of batches of messages with an efficient batching format. A batch of messages can be grouped together, compressed and sent to the server in this form.

The broker decompresses the batch to validate it. For example, the broker validates that the batch contains the same number of records that the batch header says it does. After validation, the batch of messages is written to disk in compressed form. The batch then remains compressed in the log and is transmitted to the consumer in compressed form. The consumer decompresses any compressed data that it receives.

Kafka supports GZIP, Snappy, LZ4 and ZStandard compression protocols. More details can be found on the [Compression topic on the Kafka wiki](https://cwiki.apache.org/confluence/display/KAFKA/Compression).

## explanations

Выделенный фрагмент объясняет, как Apache Kafka борется с **излишним копированием данных в памяти**, чтобы обеспечить максимальную скорость работы при высоких нагрузках.

Вот основные технические идеи, описанные в этом разделе:

- **Единый формат сообщений:** При высоких нагрузках постоянное копирование и преобразование данных сильно тормозят систему. Чтобы этого избежать, отправители (producer), серверы (broker) и получатели (consumer) в Kafka используют один и тот же стандартизированный бинарный формат. Блоки данных передаются по цепочке "как есть", без каких-либо изменений.
    
- **Оптимизация «Zero-copy» (Нулевое копирование):** В обычных системах, прежде чем отправить данные по сети, они многократно копируются туда-сюда между ядром операционной системы и памятью самого приложения (user-space). Kafka обходит эту проблему, используя специальную функцию современных ОС (Unix/Linux) — системный вызов `sendfile`.
    
- **Использование кэша страниц (Pagecache):** Благодаря `sendfile`, данные копируются в кэш операционной системы всего один раз. Когда разные приложения запрашивают эти данные, ОС отправляет их в сеть напрямую из кэша, минуя память приложения Kafka.
    
- **Скорость сети как единственный предел:** Отсутствие лишнего копирования позволяет Kafka передавать сообщения с невероятной скоростью, которая ограничивается только пропускной способностью вашего сетевого канала.
    
- **Чтение без нагрузки на диск:** В конце фрагмента отмечается интересная особенность: если получатели успевают читать данные в реальном времени (без сильного отставания), они получают их целиком из кэша ОС. Это значит, что жесткие диски серверов в этот момент вообще не нагружаются операциями чтения.
    

**Вкратце:** Kafka избегает лишней работы по перекладыванию байтов внутри оперативной памяти. Она сохраняет данные в кэш ОС один раз и напрямую "прокачивает" их в сетевой адаптер для всех потребителей, что делает систему невероятно быстрой.


Выделенный текст объясняет, как Apache Kafka использует **пакетную обработку операций ввода-вывода (I/O)** для многократного увеличения производительности системы.

Вот основные идеи, описанные в этом фрагменте:

- **Проблема мелких операций:** Слишком большое количество мелких операций ввода-вывода (как при передаче по сети между клиентом и сервером, так и при записи данных на диск) сильно замедляет работу. Отправка каждого сообщения по отдельности создает огромные накладные расходы.
    
- **Группировка сообщений (Message set):** Чтобы избежать этой проблемы, протокол Kafka построен на абстракции "наборов сообщений". Сообщения естественным образом объединяются в группы (батчи/пакеты).
    
- **Оптимизация на всех этапах:**
    
    - **Сеть:** Сообщения отправляются группами, что позволяет распределить затраты времени на сетевые пересылки (roundtrips).
        
    - **Запись на диск:** Сервер Kafka добавляет в свой журнал на жестком диске не по одному сообщению, а сразу целыми блоками (chunks) за один раз.
        
    - **Чтение:** Потребители (consumers) также запрашивают и получают данные большими линейными порциями.
        
- **Колоссальный прирост скорости:** Эта относительно простая оптимизация ускоряет работу системы на порядки (в десятки или сотни раз).
    
- **Технические плюсы батчинга:** Группировка приводит к формированию более крупных сетевых пакетов, позволяет выполнять масштабные _последовательные_ дисковые операции и использовать непрерывные блоки памяти.
    

**Вкратце:** Благодаря пакетной обработке, Kafka превращает неравномерный (импульсивный) и хаотичный поток мелких сообщений в ровный, последовательный поток данных, который максимально быстро и эффективно записывается на диск и доставляется потребителям.


Выделенный фрагмент объясняет концепцию **сквозного (end-to-end) сжатия пакетов данных** в Apache Kafka, которая помогает справиться с ограничениями пропускной способности сети.

Вот основные идеи, описанные в этом разделе:

- **Проблема сетевого трафика:** Иногда главным "узким местом" системы становится не нагрузка на процессор или диск, а пропускная способность сети (особенно при пересылке данных между разными дата-центрами через интернет).
    
- **Неэффективность поштучного сжатия:** Пользователь может сжимать каждое сообщение по отдельности до отправки в Kafka, но это дает плохой результат. В однотипных сообщениях (например, в логах или JSON-файлах) часто повторяются одни и те же заголовки или названия полей. Поштучное сжатие не может эффективно сжать эти повторы.
    
- **Сжатие целыми пакетами (батчами):** Для высокой эффективности Kafka группирует множество сообщений в один пакет (batch) и сжимает их вместе.
    
- **Как работает сквозной процесс:**
    
    1. **Отправка:** Отправитель собирает сообщения в пакет, сжимает его и отправляет.
        
    2. **Проверка:** Сервер Kafka (брокер) частично распаковывает пакет только для того, чтобы убедиться в его корректности (например, проверяет, совпадает ли реальное количество сообщений с тем, что указано в заголовке).
        
    3. **Хранение:** После проверки сервер записывает пакет на жесткий диск **в сжатом виде**.
        
    4. **Доставка:** Когда потребитель (consumer) запрашивает данные, пакет передается ему по сети также в сжатом виде.
        
    5. **Распаковка:** И только конечный потребитель полностью распаковывает полученные данные.
        
- **Алгоритмы:** Kafka "из коробки" поддерживает популярные и эффективные протоколы сжатия: GZIP, Snappy, LZ4 и ZStandard.
    

**Вкратце:** Чтобы сэкономить сетевой трафик и место на диске, Kafka сжимает не отдельные сообщения, а целые их группы. Эти группы передаются и хранятся в сжатом виде на протяжении почти всего пути от отправителя к конечному получателю, что делает работу с большими объемами данных максимально эффективной.



Это отличный вопрос! Логика работы Kafka в этом сценарии как раз и позволяет ей оставаться сверхбыстрой: **брокер (сервер) вообще не распаковывает пакет**, чтобы выдать сообщение из его середины. Вся магия происходит за счет умного использования индексов и разделения обязанностей между сервером и клиентом.

Процесс устроен следующим образом:

### 1. Поиск пакета через разреженный индекс

Для каждого файла лога (`.log`) у Kafka есть сопутствующий индексный файл (`.index`). Этот индекс является разреженным (sparse index) — он хранит позиции не каждого сообщения, а только некоторых из них (обычно каждые несколько килобайт).

Что важно: индекс всегда указывает на **начало сжатого пакета** (на так называемый _Base Offset_ — офсет самого первого сообщения в этом батче).

### 2. Брокер отправляет пакет целиком (без распаковки)

Когда потребитель (consumer) запрашивает данные, например, начиная с офсета **15**, а этот офсет находится внутри сжатого пакета, где лежат сообщения с **10** по **20**:

1. Брокер смотрит в индекс и находит физическую позицию начала пакета (с офсетом 10).
    
2. Брокер считывает **весь сжатый пакет целиком** из кэша ОС или диска.
    
3. Отправляет этот сжатый пакет потребителю по сети «как есть». Сервер принципиально не тратит ресурсы своего CPU на то, чтобы распаковать данные и вырезать из них кусок.
    

### 3. Распаковка и фильтрация на стороне Consumer

Вся вычислительная нагрузка по распаковке переносится на конечного клиента:

1. Потребитель получает сжатый пакет (в котором упакованы сообщения 10–20).
    
2. На своей стороне распаковывает его.
    
3. Пробегается по сообщениям, видит, что первые пять штук (с 10 по 14) ему не нужны (так как он просил с 15-го), **просто отбрасывает их** и начинает обрабатывать данные ровно с офсета 15.
    

---

> **Почему это эффективная архитектура?**
> 
> Если бы брокер сам распаковывал пакеты для тысяч разных потребителей, процессор сервера мгновенно бы перегрузился. Перенос распаковки и фильтрации на сторону Consumer распределяет эту нагрузку по клиентским машинам. Брокер же остается максимально разгруженным и занимается только своей главной задачей — линейным чтением и передачей байтов.


В Apache Kafka количество сообщений в группе (батче) **не задается жестким числом** (например, «ровно по 100 штук»). Вместо этого Kafka работает более гибко: размер группы определяется **объемом данных в байтах** и **временем ожидания**.

Вся магия пакетирования происходит на стороне отправителя (**Producer**). Ниже подробно разобрано, как это настраивается и как правильно это готовить в продакшене.

## Как регулируется размер группы?

Продюсер собирает сообщения в батчи отдельно для каждой партиции (раздела). Этот процесс управляется двумя главными параметрами:

- **`batch.size`** (в байтах): Это максимальный объем памяти, выделенный под один батч. Как только накопилось сообщений на эту сумму байт, батч немедленно запечатывается и отправляется на брокер.
    
    - _Дефолт:_ `16384` (16 КБ).
        
- **`linger.ms`** (в миллисекундах): Это искусственная задержка («время раздумий»). Продюсер ждет указанное количество времени, давая возможность новым сообщениям успеть упаковаться в текущий батч.
    
    - _Дефолт:_ `0` (отправлять сразу же, как только поток готов, что при низкой нагрузке сводит батчинг к нулю).
        

### Как они работают вместе?

Батч отправляется брокеру, как только выполнится **первое из двух условий**: либо наберется вес `batch.size`, либо истечет время `linger.ms`.

Поэтому итоговое _количество_ сообщений в группе — величина динамическая. Если у вас сообщения по 100 байт, в дефолтный батч 16 КБ поместится около 160 штук. Если сообщения по 5 КБ — всего 3 штуки.

## Best Practices и рекомендации по настройке

Дефолтные настройки Kafka (`batch.size=16KB`, `linger.ms=0`) заточены под **минимальную задержку (low latency)**, но они ужасны для **высокой пропускной способности (high throughput)**.

Если вы хотите выжать из Kafka максимум эффективности, используйте следующие рекомендации:

### 1. Поднимите `linger.ms` (даже чуть-чуть)

Если у вас идет плотный поток данных, установите `linger.ms` в районе `5`–`20` мс.

> **Что это даст:** Вы добавите мизерную задержку в несколько миллисекунд, но взамен продюсер начнет собирать огромные, плотные пачки сообщений вместо того, чтобы спамить брокер одиночными запросами. Нагрузка на CPU брокера упадет в разы.

### 2. Увеличьте `batch.size`

Современные 16 КБ по умолчанию — это очень мало для нагруженных систем. Если ваши батчи закрываются слишком быстро (по размеру, а не по времени), увеличивайте этот лимит.

- Для средних нагрузок: **64 КБ** (`65536`) или **128 КБ** (`131072`).
    
- Для гигантских потоков данных (логи, метрики): **512 КБ** или даже **1 МБ**.
    

### 3. Не забывайте про `buffer.memory`

Продюсер хранит еще не отправленные батчи в буфере в памяти. Дефолт — 32 МБ (`33554432`).

- Если вы увеличили `batch.size` до 512 КБ, а у вас 100 партиций, то только на открытые батчи может уйти $100 \times 512 \text{ КБ} = 50 \text{ МБ}$. Продюсер начнет блокировать отправку, так как ему не хватит буфера.
    
- **Рекомендация:** Если увеличиваете размеры батчей или работаете с сотнями партиций на одном продюсере, пропорционально поднимайте `buffer.memory` (например, до 64 или 128 МБ).
    

### 4. Включайте сжатие только вместе с батчингом

Как мы уже выяснили ранее, сжатие (`compression.type=lz4` или `zstd`) работает эффективно только на больших объемах данных. Если `linger.ms=0`, сжатие почти не даст эффекта и просто вхолостую сожжет процессор. Хороший батчинг + сжатие дают колоссальную экономию сети.

### 5. Помните про ограничения Брокера

Если вы выкрутили `batch.size` до 2 МБ, убедитесь, что брокер вообще примет такой пакет. За это отвечают настройки:

- На стороне продюсера: `max.request.size` (максимальный размер всего запроса, дефолт 1 МБ).
    
- На стороне брокера: `message.max.bytes` (максимальный размер батча, который примет сервер, дефолт 1 МБ).
    
- _Они должны быть согласованы, иначе продюсер будет ловить RecordTooLargeException._

## 1. Как работает Acknowledgment (подтверждение) при батчинге

Когда сообщения объединяются в батч, они упаковываются в один сетевой запрос (`ProduceRequest`). Настройки подтверждения (**`acks`**) работают **на уровне всего этого запроса (батча), а не отдельных сообщений**.

Брокер отправляет продюсеру ответ (`ProduceResponse`) только тогда, когда весь батч целиком прошел необходимую стадию записи:

- **`acks=0`**: Продюсер отправляет батч по сети и сразу считает его успешно доставленным. Он не ждет ответа от брокера. Если сеть моргнула или брокер упал — данные потеряны.
    
- **`acks=1`** (дефолт): Брокер принимает сжатый батч, распаковывает заголовки для проверки, записывает весь батч в свой локальный лог (в кэш ОС) и только после этого отправляет продюсеру подтверждение: _«Пакет принял, всё ок»_.
    
- **`acks=all` (или `-1`)**: Брокер запишет батч себе, разошлет его копии на реплики (фактически, другие брокеры тоже получат этот сжатый батч) и только после того, как минимально необходимое число реплик (`min.insync.replicas`) подтвердит запись, продюсер получит заветный «ок».
    

> **Главный вывод:** Для Kafka батч — это неделимая транзакционная единица на уровне сетевого обмена. Брокер не может подтвердить прием «половины батча».

## 2. Что будет, если приложение (продюсер) упало посреди отправки батча?

Здесь сценарий зависит от того, что именно произошло: сетевой сбой (приложение живо) или смерть процесса (crash).

### Сценарий А: Сетевой сбой или "упало соединение" (Приложение ЖИВО)

Допустим, продюсер начал отправлять батч размером 1 МБ. Успело уйти 400 КБ, и сеть оборвалась.

1. **На стороне брокера:** Брокер получает незавершенный поток байт. Он понимает, что размер данных в TCP-пакете не соответствует заявленному в заголовке запроса, либо просто срабатывает таймаут соединения. Брокер **полностью уничтожает и игнорирует эти 400 КБ**. В лог на диск не пишется ничего.
    
2. **На стороне продюсера:** Продюсер фиксирует сетевую ошибку (или таймаут) и, используя механизм ретраев (`retries`), **пытается отправить весь этот 1 МБ батч заново** целиком.
    

### Сценарий Б: Приложение "убито" полностью (`kill -9`, выключили по питанию)

Если процесс продюсера физически перестал существовать в момент отправки:

1. **На стороне брокера:** Происходит то же самое. Брокер видит оборванное TCP-соединение и незавершенный запрос. Он **выбрасывает** недополученные байты. На диске брокера этих сообщений не появится.
    
2. **На стороне приложения:** Так как процесс умер, вся оперативная память продюсера очистилась. Тот батч, который лежал в буфере `buffer.memory` и пытался отправиться, **потерян навсегда** (если только ваше приложение перед отправкой в Kafka параллельно не сохраняло эти данные в свою локальную БД вроде PostgreSQL или RocksDB).

## Как Kafka защищает от дубликатов при переотправке батчей?

Из Сценария А вытекает классическая проблема распределенных систем: _что если брокер получил батч полностью, записал его, но в этот момент упала сеть, и подтверждение (ack) не дошло до продюсера?_ Продюсер ведь подумает, что батч потерялся, и отправит его снова.

Для защиты от этого в Kafka есть **Идемпотентность продюсера** (`enable.idempotence=true`, по умолчанию включена во всех современных версиях):

1. Каждому продюсеру при старте брокер выдает уникальный идентификатор — **PID** (Producer ID).
    
2. Каждому отправляемому батчу внутри продюсера присваивается строго последовательный порядковый номер — **Sequence Number**.
    
3. Брокер хранит на диске карту последних записанных номеров для каждого PID. Если из-за сетевого сбоя продюсер пришлет тот же самый батч повторно, брокер посмотрит на Sequence Number, скажет: _«А, я это уже записал»_, проигнорирует запись на диск, но **повторно отправит продюсеру подтверждение (ack)**, чтобы тот со спокойной душой очистил свою память.



И то, и другое! **`retries` — это официальный параметр конфигурации продюсера, но в современных версиях Apache Kafka он работает автоматически «из коробки»**.

Вам больше не нужно настраивать его вручную для базовой защиты от сбоев, но полезно понимать, как этот механизм устроен под капотом.

## Как это работает по умолчанию (автоматически)

В старых версиях Kafka дефолтное значение `retries` равнялось нулю, и разработчикам приходилось включать его руками. Сейчас всё изменилось:

- **Количество попыток по умолчанию:** `2147483647` (максимальное значение Integer, то есть фактически **бесконечно**).
    
- **Пауза между попытками (`retry.backoff.ms`):** по умолчанию **100 мс**. Если батч не ушел, продюсер подождет 100 мс и попробует снова.
    

### Если попыток бесконечно много, почему приложение не зависнет навсегда?

Продюсер не будет пытаться отправить батч до скончания веков, потому что бесконечные ретраи ограничиваются другим важнейшим автоматическим параметром — **`delivery.timeout.ms`**.

- По умолчанию он равен **120 000 мс (2 минуты)**.
    
- Этот таймер запускается в тот момент, когда батч встает в очередь на отправку.
    
- Продюсер будет циклически переотправлять батч каждые 100 миллисекунд, но если за 2 минуты пакет так и не доберется до брокера (например, жестко упала сеть), продюсер сдастся, удалит батч из памяти и выдаст приложению ошибку (`TimeoutException`).

## Когда имеет смысл менять конфигурацию вручную?

Хотя всё работает автоматически, дефолтные настройки меняют в двух основных сценариях:

### 1. Жесткое ограничение по времени (Low Latency)

Если для вашего бизнеса данные актуальны только здесь и сейчас (например, котировки акций или координаты такси в реальном времени), вам не нужно, чтобы продюсер пытался отправить старый батч целых 2 минуты.

- _Что делают:_ Уменьшают параметр `delivery.timeout.ms` (например, до `5000` — 5 секунд). Параметр `retries` при этом обычно оставляют дефолтным (бесконечным), чтобы за эти 5 секунд продюсер успел сделать как можно больше быстрых попыток.
    

### 2. Управление порядком сообщений (`max.in.flight.requests.per.connection`)

Если продюсер отправляет батч №1 и батч №2 одновременно, батч №1 может упасть с сетевой ошибкой и уйти на ретрай, а батч №2 запишется успешно. В итоге на брокере батч №2 окажется раньше, чем батч №1 — порядок нарушится.

- _Как это решено автоматически:_ В современных версиях Kafka включена **идемпотентность** (`enable.idempotence=true`). Она гарантирует сохранение строгого порядка сообщений даже при ретраях, если параметр `max.in.flight.requests.per.connection` (кол-во параллельных запросов в одну сессию) не превышает **5**.
    
- Если же вы сознательно отключаете идемпотентность (что делать не рекомендуется), но вам важен порядок, вам придется выставить `retries=0` (запретить ретраи вовсе) или ограничить параллельные запросы до 1.
