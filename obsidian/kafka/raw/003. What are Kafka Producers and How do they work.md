### Understanding Apache Kafka

**0:15** · To understand what a producer is, and how it fits into the overall picture of creating a real-time data streaming platform let’s talk a little bit about Apache Kafka.

**0:25** · Apache Kafka is a distributed streaming platform that provides a publish-subscribe messaging system.

**0:30** · In Kafka, producers are responsible for publishing messages to a Kafka topic.

**0:35** · Each topic is spread across multiple partitions.

**0:39** · These partitions help Kafka scale by allowing you to add resources and additional partitions as your system handles more and more data.

**0:47** · Each partition is an ordered, immutable sequence of messages that is continually appended to.

**0:53** · Think of a Kafka topic as a library and each partition as a different bookshelf in that library.

**0:58** · Just as different bookshelves contain different books, each partition in a topic holds a subset of the overall data.

**1:05** · So, that makes each book in a library an individual message that is sent to Kafka.

**1:10** · Each of these messages contain three elements, a key, the value of the message, and a timestamp of when the message was produced.

**1:19** · If you provide a key with the message, the producer first computes a hash on the key.

**1:23** · It then applies a mod operation with the total number of partitions to find the partition to produce to.

**1:30** · This allows messages with the same key to be written to the same partition and in the correct order.

**1:36** · If your key is null, a sticky partition is used.

**1:39** · This essentially means that messages are batched either by a certain size or a period of time and then sent to the same partition.

**1:47** · At which point a new batch will be started and sent to a different partition.

**1:51** · This helps drive latency down while still spreading messages out over all of the partitions.

**1:57** · Producers in Kafka can be implemented in just about any programming language, whether officially supported or through various community projects.

**2:05** · No matter which language you select, the best way for producers to send messages to Kafka topics is to use the Producer API.

### Producer Configuration

**2:13** · Now that we have some context and know a little bit more about how partitions work, let’s take a look at what a basic Kafka producer looks like.

**2:20** · There are four main code sections in a producer: First, we specify some configuration details. This is stuff like how to connect to the Kafka cluster and the type of the key, and the value.

**2:32** · Next, we create a producer by invoking the KafkaProducer class.

**2:36** · We then create the shutdown behavior.

**2:38** · And, last of all, we actually send the data.

**2:41** · Here you create an instance of the ProducerRecord class and provide the topic, the key, and value for your messages.

**2:48** · After you've created the ProducerRecord, you pass it to the 'send' method of KafkaProducer, it is serialized and the partition on which it will land is determined.

**2:58** · As mentioned earlier, if you’ve provided a key, all the messages will be sent to the same partition.

**3:03** · This is so that when you read the messages back using a consumer, you have the guarantee that they are in order.

**3:09** · If you didn’t provide a key, each message will be divided up in batches among all of your partitions.

**3:15** · Each partition has a “leader” node handling all its read and write requests, and “follower” nodes that replicate the data for fault tolerance and high availability.

**3:24** · You can run a cluster without having followers, but you run the risk of losing data.

**3:29** · Leaders and followers help in fault tolerance and data replication. These partitions are spread across different brokers.

**3:36** · If one of these brokers goes down, a new leader is selected from the followers.

### Producer "Acks"

**3:41** · There is one more crucial decision you need to make when writing a producer.

**3:45** · How does a producer know if the messages have been delivered?

**3:48** · For that, we have producer acknowledgment or Acks for short.

**3:52** · In other words, how important is it that your messages arrive?

**3:56** · There are three options to choose from, Acks 0, Acks 1, or Acks -1.

**4:02** · Acks 0 as the name implies is none.

**4:04** · The messages are sent without the producer waiting to see if they ever arrive in your Kafka cluster.

**4:09** · This provides the lowest latency, but there is a good chance that you will lose data.

**4:14** · Acks 1 sends the message and waits for the leader to acknowledge that it has received the message.

**4:20** · This assures that your messages are delivered, without slowing the delivery dramatically.

**4:26** · The last one is Acks -1.

**4:28** · This is where your producer waits for the leader and any followers to acknowledge that the message has been delivered.

**4:34** · While the slowest of the three, this also provides the greatest assurance that your messages have arrived and have been written to multiple disks.
