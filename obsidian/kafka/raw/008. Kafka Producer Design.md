Apache Kafka® is an open-source distributed streaming system used for stream processing, real-time data pipelines, and data integration at scale.

This topic describes how Kafka producers are designed to enable load balancing and asynchronous send operations.

## What is a producer?

A producer is a client application that publishes data to a Kafka cluster. A Kafka producer works by writing messages directly to topics in a Kafka broker. The sections in this topic describe how producers are designed to enable load balancing and asynchronous send operations.

## Producer load balancing

The client producer controls which partition it publishes messages to. Also, the producer sends data directly to the broker that is the partition leader without any intervening routing tier. To help a producer do this, all Kafka nodes provide metadata that specifies the brokers that are alive, and the brokers that are leaders for the partitions of a topic. This enables a producer to appropriately route its requests.

Load balancing can be random, or you can apply a semantic partitioning function. You can specify a key to partition by, and Kafka uses that key to hash to a partition.

There is also an option to override the partition function if needed. For example, if the specified key is `user id` then all data for a given user would be sent to the same partition. This in turn will enables consumers to make locality assumptions about their consumption. This style of partitioning is explicitly designed to allow locality-sensitive processing in consumers.

## Batching

Batching enables efficiency, and to enable batching the Kafka producer tries to accumulate data in memory, and sends larger batches in a single request. The batching can be configured:

- By batch size (example: 64 kb). Use the [batch.size](https://docs.confluent.io/platform/current/installation/configuration/producer-configs.html#batch-size) property to set this.
- By wait time (example: 10 ms). Use the [linger.ms](https://docs.confluent.io/platform/current/installation/configuration/producer-configs.html#linger-ms) property to set this.

This batching enable the accumulation of more bytes to send, and a few larger I/O operations on the servers. It also provides a mechanism to trade a small amount of additional latency for better throughput.

- ## explains 

Выделенный раздел **«Producer load balancing» (Балансировка нагрузки на стороне отправителя)** из [документации Confluent](https://docs.confluent.io/kafka/design/producer-design.html#what-is-a-producer) объясняет, как именно Kafka распределяет сообщения по разделам (партициям) и почему эта схема работает максимально эффективно.

Вот основные идеи этого фрагмента:

- **Прямая отправка без посредников:** Продюсер (отправитель) отправляет данные напрямую тому брокеру (серверу), который является лидером (leader) для конкретной партиции. В архитектуре Kafka нет промежуточных прокси-серверов или балансировщиков (routing tier), что исключает лишние сетевые задержки.
    
- **Использование метаданных:** Чтобы продюсер знал точный адрес, все узлы Kafka постоянно предоставляют ему актуальные метаданные: информацию о том, какие серверы сейчас работают и какие партиции за какими брокерами закреплены. Продюсер сам маршрутизирует свои запросы.
    
- **Способы распределения нагрузки:** Выбор партиции полностью контролируется клиентом. Распределение может быть:
    
    - _Случайным (или Round-Robin):_ когда сообщения просто равномерно распределяются по разным партициям для баланса нагрузки.
        
    - _Семантическим (по ключу):_ вы можете связать сообщение с определенным ключом, и Kafka использует хэш-функцию от этого ключа, чтобы закрепить его за конкретной партицией.
        
- **Локальность данных (Data Locality):** Метод хэширования можно переопределить. Если в качестве ключа выбрать, например, `user_id`, то все события этого пользователя гарантированно полетят в одну и ту же партицию. Это позволяет приложениям-получателям (consumers) делать «предположения о локальности» — они могут эффективно обрабатывать данные конкретного пользователя в строгом хронологическом порядке, зная, что они не размазаны по разным серверам.
    

**Вкратце:** Kafka переносит задачу балансировки нагрузки с сервера на самого клиента (продюсера). Продюсер сам решает, в какую партицию отправить сообщение (на основе ключа или алгоритма распределения), и шлет данные напрямую нужному брокеру.

Короткий ответ: **В 95% случаев — нет, это классический антипаттерн**, который только добавит вам проблем, увеличит задержки (latency) и начнет ломать логику работы сетевого протокола Kafka.

Как раз в том фрагменте документации, который мы разбирали в начале, крылся ответ: _«producer sends data directly to the broker that is the partition leader without any intervening routing tier»_ (продюсер отправляет данные напрямую брокеру-лидеру без каких-либо промежуточных маршрутизаторов).

Вот подробный разбор, почему обычный балансировщик всё испортит, и в каких редких исключениях прокси всё-таки имеет смысл.

## Почему стандартный балансировщик (NGINX, HAProxy, AWS ALB) — это плохая идея?

### 1. Он ломает концепцию «умного клиента» (Smart Client)

Kafka изначально спроектирована так, что вся логика балансировки перенесена на сторону приложения (Producer/Consumer).

При старте приложение стучится на брокер и запрашивает **метаданные** — карту кластера. Оно узнает: _«Ага, Топик А, Партиция 0 лежит на Брокере №1, а Партиция 1 — на Брокере №2»_.

После этого продюсер открывает **прямые TCP-соединения** ко всем нужным брокерам.

Если вы поставите между ними обычный балансировщик, который будет раскидывать запросы по round-robin:

- Продюсер захочет отправить батч в Партицию 0 (которая на Брокере №1).
    
- Он шлет запрос, но балансировщик перенаправляет этот TCP-пакет на Брокер №2.
    
- Брокер №2 смотрит на запрос и говорит: _«Я не лидер для этой партиции!»_ — и возвращает ошибку `NotLeaderForPartitionException`.
    
- Ваше приложение начинает бесконечно переотправлять данные и падать по таймаутам.
    

### 2. Лишняя нагрузка и потеря Zero-Copy

Как мы разбирали ранее, Kafka эффективна за счет сквозной передачи байтов напрямую в сетевой адаптер. Если на пути встанет прокси-сервер, ему придется принимать этот гигантский поток трафика, парсить его, держать в своей памяти и переотправлять дальше. Вы получите «узкое место» (bottleneck), где этот прокси быстро упрется в потолок по CPU или пропускной способности сети.

## Когда балансировщик или прокси ВСЁ-ТАКИ нужен? (Исключения)

Существует три сценария, когда промежуточный слой оправдан, но работают они не так, как классический балансировщик веб-трафика.

### Сценарий 1: Только для начального знакомства (Bootstrap)

Вы можете поставить балансировщик перед кластером Kafka исключительно для конфигурации `bootstrap.servers`.

- Приложение коннектится к `lb.kafka.company.com:9092`.
    
- Балансировщик перенаправляет этот первый запрос на _любой_ живой брокер.
    
- Брокер возвращает клиенту метаданные, в которых написаны **реальные внешние адреса** всех брокеров кластера (за это отвечает настройка `advertised.listeners` на серверах).
    
- Получив этот список, приложение **отключается от балансировщика** и дальше общается с брокерами напрямую по их реальным IP/доменам.
    

### Сценарий 2: Сети Kubernetes и SNI-проксирование (Envoy / HAProxy)

Если ваша Kafka живет внутри Kubernetes, а приложения-клиенты находятся снаружи (в другой сети), они не могут достучаться до внутренних IP-адресов подов.

В этом случае ставят прокси (например, Envoy), но он настраивается на работу через **TLS SNI (Server Name Indication)**.

- Прокси не балансирует трафик случайно.
    
- Если приложение запрашивает `broker-1.kafka.company.com`, Envoy четко знает, что этот трафик нужно направить строго в под Брокера №1. Архитектура прямых соединений «клиент-лидер» при этом сохраняется.
    

### Сценарий 3: Ограничения безопасности или legacy-системы (Kafka REST Proxy)

Бывают ситуации, когда из соображений безопасности фаерволы запрещают открывать тысячи постоянных TCP-соединений от микросервисов к брокерам, либо приложение написано на старом языке, где нет хорошей нативной библиотеки для Kafka.

Тогда рядом с кластером ставят специальный компонент — **Kafka REST Proxy** (например, от Confluent).

- Перед REST Proxy ставится обычный балансировщик (NGINX/HAProxy).
    
- Приложения шлют стандартные, простые HTTP POST-запросы с JSON-данными на балансировщик.
    
- Балансировщик распределяет HTTP-нагрузку по серверам REST Proxy.
    
- А уже сам REST Proxy выступает в роли полноценного, "умного" Kafka-клиента и общается с брокерами напрямую по родному TCP-протоколу.
    

---

**Резюме:** Если ваши приложения могут общаться по нативному протоколу Kafka и имеют прямой сетевой доступ к серверам — **никакие балансировщики и прокси посередине не нужны**. Продюсер сам идеально распределит нагрузку.

А какая у вас сетевая топология? Приложения и Kafka находятся в одном контуре (дата-центре), или вы планируете давать доступ внешним клиентам через интернет / Kubernetes?
