## Kafka Replication and Committed Messages

Apache Kafka® is an open-source distributed streaming system used for stream processing, real-time data pipelines, and data integration at scale.

This topic describes how Kafka replication is configured and works, and how it enables committed messages.

## Replication overview

Kafka replicates the event log for each topic’s partitions across a configurable number of servers. This replication factor is configured at the topic level, and the unit of replication is the topic partition. This enables automatic failover to these replicas when a server in the cluster fails so messages remain available.

Under non-failure conditions, each partition in Kafka has a single leader and zero or more followers. In Kafka, all topics must have a replication factor configuration value. The replication factor includes the total number of replicas including the leader, which means that topics with a replication factor of one (1) are topics that are not replicated.

All reads and writes go to the leader of the partition.

Typically, there are many more partitions than brokers and the leaders are evenly distributed among brokers. The logs on the followers are identical to the leader’s log; all have the same offsets and messages in the same order. Although at any given time, the leader may have a few unreplicated messages at the end of its log.

The following image shows a topic with three partitions and how they might be replicated across three brokers.

![../../_images/replication.png](https://docs.confluent.io/_images/replication.png)

Followers consume messages from the leader like a Kafka consumer would and apply them to their own log. Followers pulling from the leader enables the follower to batch log entries applied to their log.

As with most distributed systems, automatically handling failures requires having a precise definition of what it means for a node to be *alive*. For Kafka node to be considered alive, it has to meet two conditions

1. A node must be able to maintain its session with the controller
2. If it is a follower it must replicate the writes happening on the leader and not fall “too far” behind

These nodes are called “in sync” versus “alive” or “failed”. The leader keeps track of the set of in-sync nodes. If a follower fails, gets stuck, or falls behind, the leader will remove it from the list of in-sync replicas. The `replica.lag.time.max.ms` configuration specifies what replicas are considered stuck or lagging.

In distributed systems terminology, Kafka attempts to handle “fail/recover” scenarios where nodes suddenly cease working and then later recover, but does not handle [Byzantine failures](https://en.wikipedia.org/wiki/Byzantine_fault), in which nodes produce arbitrary or malicious responses due to foul play or bugs.

## In-sync replicas and producer acks

Kafka guarantees that a committed message will not be lost, as long as there is at least one in-sync replica alive at all times. In this context, a committed message means that all in-sync replicas for a partition have applied the message to their log. Consumers will only receive committed messages, meaning a consumer will never see a message that is potentially lost if a leader fails.

To enable this functionality, a producer provides the [acks](https://docs.confluent.io/platform/current/installation/configuration/producer-configs.html#acks) property, which controls whether the producer waits for the message to be committed, balancing latency versus durability for that producer.

Use the `acks=all` setting to request acknowledgment that a message has been written to the full set of in-sync replicas. You configure how many in-sync replicas are required with the [min.insync.replicas](https://docs.confluent.io/platform/current/installation/configuration/topic-configs.html#min-insync-replicas) property at the topic level.

If a producer requires less stringent acknowledgment (`acks=0` or `acks=1`), the producer will receive the acknowledgment earlier. With `acks=0`, the producer will immediately ack without waiting for the response from the server. With `acks=1`, the server will ack to the producer if the messages are written to the leader’s local log. Then, the messages are available to the consumers after they are replicated to all the in-sync replicas (ISRs). Note that, if the number of ISRs is less than the minimum setting, the messages will not be available to the consumers until the number of ISRs has reached the minimal setting and the messages are replicated to all the ISRs.

Kafka remains available in the presence of node failures after a short fail-over period, but may not remain available in the presence of network partitions.

## Replicated logs: quorums, ISRs, and state machines

As discussed previously, a Kafka partition is a replicated log. The replicated log is one of the most basic primitives in distributed data systems, and there are many approaches for implementing one. A replicated log can be used as a primitive for implementing distributed systems in the [state-machine style](http://en.wikipedia.org/wiki/State_machine_replication).

A replicated log models the process of coming into consensus on the order of a series of values, generally numbering the log entries 0, 1, 2 and so on. The simplest way to implement this is with a leader that chooses the ordering of values provided to it. As long as the leader remains alive, all followers need to only copy the values in the order specified by the leader.

When a leader fails, a new leader must be chosen from the followers, but followers may fall behind or crash so the chosen follower must be up-to-date. The fundamental guarantee a log replication algorithm provides is that if we tell a client a message is committed, and the leader fails, the newly-elected leader must also have that message. This results in a tradeoff: if the leader waits for more followers to acknowledge a message before declaring it committed, then there are more electable leaders.

A *quorum* occurs when when you choose the number of acknowledgements required and the number of logs that must be compared to elect a leader so that an overlap is guaranteed.

One approach to this is to use a majority vote for both the commit decision and the leader election. A downside of majority vote is that the majority of nodes must be running to tolerate a failure. With majority vote, tolerating one failure requires three copies of the data, tolerating two failures requires five copies of the data.

However, majority vote has a nice property: latency depends on only the fastest servers. That is, if the replication factor is three, the latency is determined by the faster follower not the slower one.

Kafka running with ZooKeeper takes a slightly different approach to choosing its quorum set. Instead of majority vote, Kafka dynamically maintains a set of in-sync replicas (ISR) that are caught-up to the leader. Only members of this set are eligible for election as leader. A write to a Kafka partition is not considered committed until *all* in-sync replicas have received the write. This ISR set is persisted to Zookeeper whenever it changes, and any replica in the ISR is eligible to be elected leader. This is an important factor for Kafka’s usage model where there are many partitions and ensuring leadership balance is important. With this ISR model and *f+1* replicas, a Kafka topic can tolerate *f* failures without losing committed messages.

For most use cases, this tradeoff is a reasonable one. In practice, to tolerate *f* failures, both the majority vote and ISR approach will wait for the same number of replicas to acknowledge before committing a message. For example, to survive one failure:

- A majority vote quorum needs three replicas and one acknowledgement
- The ISR approach requires two replicas and one acknowledgement

The ability to commit without the slowest servers is an advantage of the majority vote approach.

Another important design distinction is that Kafka does not require that crashed nodes recover with all their data intact. Often replication algorithms depend on the existence of “stable storage” that cannot be lost in any failure-recovery scenario without potential consistency violations. There are two primary problems with this assumption.

- Disk errors are the most common problem observed in the real operation of persistent data systems and they often do not leave data intact.
- Requiring the use of fsync on every write for consistency guarantees can reduce performance by two to three orders of magnitude. The Kafka protocol for allowing a replica to rejoin the ISR ensures that before rejoining, it must fully re-sync again even if it lost unflushed data in its crash.

## Unclean leader election and partition loss

Note that Kafka’s guarantee for data loss relies on at least one replica remaining in sync. If all the nodes replicating a partition die, this guarantee is invalidated.

If all of the replicas die, there are two possible behaviors:

1. Wait for a replica in the ISR to come back to life and choose this replica as the leader with the hope it retains all of its data.
2. Choose the first replica (not necessarily in the ISR) that comes back to life as the leader.

This is a simple tradeoff between availability and consistency. If the system waits for replicas in the ISR, then it will remain unavailable as long as those replicas are down. If the ISR replicas were destroyed or their data was lost, the system is permanently down. If, on the other hand, a non-in-sync replica comes back to life and it becomes the leader, then its log becomes the source of truth even though it is not guaranteed to have every committed message. By default from version 0.11.0.0, Kafka chooses the first strategy and favors waiting for a consistent replica. This behavior can be changed using configuration property `unclean.leader.election.enable`, to support use cases where uptime is preferable to consistency.

This tradeoff is not specific to Kafka, and exists in any quorum-based scheme. For example in a majority voting quorum, if a majority of servers suffer permanent failures, then you must either choose to lose 100% of your data or violate consistency by taking what remains on an existing server as your new source of truth.

## Replica management

The previous section on replicated logs covers a single log, in other words, one topic partition. However a Kafka cluster manages hundreds or thousands of these partitions. Kafka attempts to balance partitions within a cluster in a round-robin fashion. This avoids clustering all partitions for high-volume topics on a small number of nodes. Likewise, Kafka also tries to balance leadership so that each node is the leader for a proportional share of its partitions.

The leadership election process is the critical window of unavailability, and must be optimized. To accomplish this, one of the brokers is elected as the “controller”. This broker controller detects failures at the broker level and changes the leader of all affected partitions when a broker fails. The result is leadership change notifications are batched, which makes the election process fast for a large number of partitions. If the controller fails, one of the surviving brokers will become the new controller.

## 1. Объяснение концепций (Explanation)

Выделенный текст описывает базовую архитектуру и принципы отказоустойчивости Kafka, основанные на репликации данных:

- **Единица репликации:** Копирование лога событий происходит на уровне **партиций (разделов)** топика, а не топиков целиком. Количество копий задается параметром _Replication Factor_. Фактор, равный 1, означает полное отсутствие репликации.
    
- **Ролевая модель (Leader/Follower):** У каждой партиции всегда есть один **Лидер (Leader)** и ноль или более **Фолловеров (Followers)**. Все операции записи и чтения от клиентов жестко идут только через Лидера.
    
- **Механизм работы фолловеров:** Фолловеры не принимают трафик от клиентов. Они работают как обычные потребители (consumers) — постоянно отправляют _pull-запросы_ лидеру, скачивают новые сообщения пакетами (батчами) и сохраняют их в свой локальный лог в идентичном порядке.
    
- **Синхронность и статус ISR (In-Sync Replicas):** Брокер считается живым и «синхронным», если он удерживает сессию с контроллером кластера и не отстает от лога лидера слишком сильно. Время допустимого отставания регулируется параметром `replica.lag.time.max.ms`. Лидер постоянно ведет список таких актуальных узлов (ISR).
    
- **Модель отказов:** Архитектура Kafka спроектирована под сценарии типа _fail/recover_ (когда узел аварийно остановился, а затем восстановился с сохранением структуры данных). Система **не рассчитана** на византийские ошибки (Byzantine failures), вызванные умышленной порчей данных или критическими багами, генерирующими случайные ответы.

## 2. Лучшие практики (Best Practices)

Для построения надежной распределенной системы на базе описанных принципов рекомендуется применять следующие стандарты:

- **Фактор репликации для Production:** Используйте `replication.factor=3` для всех бизнес-критичных топиков. Это гарантирует, что система переживет одновременный выход из строя одного брокера без потери доступности на запись, и до двух брокеров — на чтение.
    
- **Осведомленность о стойках (Rack Awareness):** Обязательно настраивайте параметр `broker.rack` в конфигурации брокеров. Указывайте туда имя физической стойки, блейд-шасси или зоны доступности (например, `us-east-1a`). Kafka автоматически распределит реплики одной партиции по разным дата-центрам/стойкам, защищая от масштабных аварий инфраструктуры.
    
- **Автоматическая балансировка лидеров:** Оставляйте включенным параметр `auto.leader.rebalance.enable=true`. Это позволяет кластеру автоматически возвращать лидерство «родным» брокерам (Preferred Leaders) после их успешного восстановления, равномерно распределяя сетевую нагрузку по всему кластеру.

## 3. Реальные кейсы использования (Case Studies)

- **Облачные инфраструктуры (Multi-AZ Deployment):** Такие компании, как Netflix и Uber, разворачивают кластеры Kafka в нескольких изолированных зонах доступности (Availability Zones). Благодаря репликации по модели ISR, даже в случае полного падения одной из зон AWS/GCP, лидерство мгновенно переходит к фолловерам в выживших зонах. Клиентские приложения замечают лишь мимолетный скачок задержки (в пределах нескольких секунд).
    
- **Финтех и платежные шлюзы (Zero Data Loss):** В банковских системах репликация выступает гарантом сохранности транзакций. За счет работы строго через лидера и копирования в ISR, система защищена от «фантомных» записей. Если сервер с лидером сгорает, новым лидером может стать только узел, который уже подтвердил запись данного платежа, что исключает расхождение балансов.

## 4. Рекомендации для продакшена (Production-Ready Recommendations)

Для обеспечения стабильности, производительности и масштабируемости под нагрузкой используйте следующие технические настройки:

### Настройки коммитов и реплик

- **Связка параметров надежности:** Устанавливайте `min.insync.replicas=2` на уровне топиков при `replication.factor=3`. В сочетании с продюсерским параметром `acks=all` это не позволит записать данные, если в кластере жива всего одна копия партиции, предотвращая появление нереплицированных сообщений, которые могут потеряться при аварии.
    

### Тюнинг тайм-аутов

- **Оптимизация `replica.lag.time.max.ms`:** По умолчанию этот параметр равен 30 000 мс (30 секунд). Для высоконагруженных систем со стабильной сетью между брокерами рекомендуется уменьшить его до `10000`–`15000` (10–15 секунд). Это позволит системе значительно быстрее выводить зависшие или перегруженные дисковым вводом-выводом (I/O) брокеры из списка ISR, не заставляя продюсеров висеть в таймаутах.
    

### Сетевая оптимизация потока репликации

- **Параллельные потоки репликации:** Если в кластере созданы тысячи партиций, стандартный одиночный поток репликации может не успевать вычитывать данные у лидера. Увеличьте количество потоков-выборщиков с помощью параметра `num.replica.fetchers` (оптимально: от 2 до 4, в зависимости от количества ядер CPU на брокерах).
    
- **Контроль размера батча репликации:** Настройте `replica.fetch.max.bytes` (размер данных, вытягиваемых фолловером за один запрос) так, чтобы он был равен или немного превышал максимальный размер сообщения в вашем кластере (`message.max.bytes`), предотвращая зависание репликации на крупных объектах.

### 1. Объяснение концепций (Explanation)

В данном разделе объясняется, как контролируется коммит сообщений в Kafka и как настройки продюсера позволяют балансировать между скоростью работы и надежностью хранения данных:

- **Понятие закоммиченного сообщения:** Сообщение считается официально закоммиченным (committed) только тогда, когда **все** работающие в данный момент синхронные реплики (In-Sync Replicas — ISR) записали его в свои логи.
    
- **Гарантия видимости:** Потребители (consumers) имеют право читать только закоммиченные сообщения. Это гарантирует, что клиент никогда не увидит запись, которая может бесследно исчезнуть в случае внезапного падения брокера-лидера.
    
- **Параметр `acks` (Подтверждения):** Продюсер использует это свойство для управления уровнем надежности:
    
    - `acks=all` (или `-1`): Максимальная надежность. Продюсер ждет, пока сообщение запишется на весь набор ISR. Минимально допустимое число таких реплик для успешной записи конфигурируется параметром `min.insync.replicas`.
        
    - `acks=1`: Средняя надежность. Продюсер получает подтверждение сразу, как только лидер партиции записал данные локально. При этом потребители все равно не увидят сообщение, пока оно не скопируется в остальные реплики ISR. Если количество живых ISR меньше минимума, сообщения станут доступны консьюмерам только после восстановления состава группы репликации.
        
    - `acks=0`: Минимальная задержка. Продюсер отправляет данные в сеть и сразу считает их успешными, не дожидаясь ответа от сервера.
        
- **Поведение при сетевом разделении:** Kafka устойчива к отказам отдельных узлов (fail-over происходит быстро), однако в условиях сетевого распада (network partitions) доступность кластера может быть временно ограничена для предотвращения рассинхронизации.

### 2. Лучшие практики (Best Practices)

- **Строгая классификация данных:** Четко разделяйте потоки данных. Для критически важных бизнес-событий (транзакции, изменения баланса, заказы) всегда форсируйте `acks=all`. Для логов, кликстрима или метрик, где потеря нескольких записей не критична, переходите на `acks=1` для разгрузки сети.
    
- **Защита от «одинокого лидера»:** Никогда не используйте `acks=all` без явной настройки `min.insync.replicas`. Если фактор репликации топика равен 3, установите минимум синхронных реплик на уровне 2. Это предотвратит ситуацию, когда 2 брокера упали, лидер остался один, и система продолжает принимать «надежные» (по мнению продюсера) записи, которые физически никуда не дублируются.
    
- **Включение идемпотентности продюсера:** При использовании `acks=all` обязательно включайте параметр `enable.idempotence=true`. При сетевых сбоях или таймаутах продюсер может повторно отправлять сообщения. Идемпотентность гарантирует, что брокер отсечет дубли, и в лог запишется ровно одна копия.

### 3. Кейсы использования (Case Studies)

- **Процессинг платежей и финтех:** При обработке банковских переводов критически важно избежать потери сведений о транзакции. Использование комбинации `acks=all` + `min.insync.replicas=2` гарантирует, что если брокер-лидер выйдет из строя (например, из-за сбоя по питанию) сразу после фиксации перевода, транзакция останется невредимой на репликах, и новый лидер продолжит работу без потери целостности данных.
    
- **Системы логистики и цепочки поставок:** Обновление статусов доставки или складских остатков требует согласованности. Использование режима `acks=all` исключает появление фантомных записей («ghost reads»), когда консьюмер успел бы прочитать промежуточный статус, который затем исчез из-за сбоя нереплицированного лидера.
    
- **Высокоскоростной сбор метрик и IoT (Противоположный кейс):** При сборе данных с тысяч датчиков температуры или GPS-трекеров задержка сети критичнее потери пакета. Здесь успешно применяется `acks=0` или `acks=1`, что позволяет обрабатывать миллионы событий в секунду за счет отказа от синхронного ожидания репликации.

### 4. Рекомендации для продакшена (Production-Ready Recommendations)

- **Конфигурация "Золотого стандарта" для важных топиков:**
    
    Ini, TOML
    
    ```
    # На уровне топика/брокера
    replication.factor=3
    min.insync.replicas=2
    unclean.leader.election.enable=false
    
    # На уровне продюсера
    acks=all
    enable.idempotence=true
    retries=2147483647 (или MAX)
    ```
    
- **Корректная обработка исключений на клиенте:** При падении количества реплик ниже лимита продюсер начнет получать ошибку `NotEnoughReplicasException`. Ваше приложение должно уметь корректно обрабатывать это исключение (например, отправлять сообщения во временный локальный буфер или приостанавливать чтение входящего трафика), чтобы не терять данные в памяти.
    
- **Тюнинг задержки репликации:** Сократите значение `replica.lag.time.max.ms` до 10000–15000 мс (по умолчанию 30 секунд). В продакшене под высокой нагрузкой 30 секунд — это слишком долгий срок для выявления зависшего или отстающего брокера. Быстрое исключение проблемного узла из ISR стабилизирует работу продюсеров с `acks=all`.
    
- **Изоляция инфраструктуры (Rack Awareness):** Распределяйте брокеры по разным физическим стойкам или зонам доступности (Availability Zones) с помощью параметра `broker.rack`. Это минимизирует риски полной потери топика при локальных сетевых разделениях (network partitions) внутри одного дата-центра.



#### Replicated logs: quorums, ISRs, and state machines
Анализ раздела **«Replicated logs: quorums, ISRs, and state machines» (Реплицируемые логи: кворумы, ISR и машины состояний)** из документации [Kafka Replication](https://docs.confluent.io/kafka/design/replication.html#replicated-logs-quorums-isrs-and-state-machines).

---

## 1. Объяснение концепций (Explanation)

В этом фрагменте подробно описывается математическая и концептуальная модель, на которой держится отказоустойчивость Kafka, а также её ключевое отличие от традиционных распределенных систем:

- **Лог как машина состояний:** Партиция Kafka — это упорядоченный набор записей (0, 1, 2...). Лидер определяет этот порядок, а фолловеры просто копируют его. Главная задача алгоритма — гарантировать, что если клиенту подтвердили запись сообщения (оно закоммичено), это сообщение никогда не исчезнет при смене лидера.
    
- **Дилемма Кворума (Большинство против ISR):**
    
    - **Классическое большинство (Majority Vote):** Чтобы пережить потерю `f` серверов, системе нужно иметь `2f + 1` реплик (например, для защиты от 1 сбоя нужно 3 копии). Плюс: скорость записи зависит от _самых быстрых_ узлов. Минус: огромный оверхед по дискам.
        
    - **Модель ISR (In-Sync Replicas) в Kafka:** Вместо жесткого голосования большинства, Kafka динамически собирает группу «успевающих реплик» (ISR). Сообщение считается закоммиченным, только когда его получили **все** участники ISR. Для защиты от `f` сбоев требуется всего `f + 1` реплик (для защиты от 1 сбоя достаточно 2 копий). Минус: скорость коммита ограничена _самым медленным_ узлом внутри ISR.
        
- **Отказ от обязательного `fsync` (Stable Storage):** Большинство алгоритмов требуют сбрасывать данные на диск (`fsync`) при каждой записи, что замедляет систему в сотни раз. Kafka идет другим путем: она разрешает узлам хранить данные в оперативной памяти (Page Cache). Если брокер упал и потерял незафлашенные данные, протокол Kafka просто запретит ему возвращаться в ISR, пока он полностью не пересинхронизирует свой лог с живым лидером с нуля.
    

---

## 2. Лучшие практики (Best Practices)

- **Оптимальный баланс стоимости и надежности:** Используйте стандартную формулу `replication.factor=3` совместно с `min.insync.replicas=2`. Это идеальное воплощение преимуществ модели ISR: вы тратите ресурсы всего на 3 копии данных, но при этом гарантированно защищены от падения одного любого брокера.
    
- **Доверьтесь ОС и не трогайте настройки флаша:** Не пытайтесь принудительно настраивать параметры `flush.messages` или `flush.ms`. Kafka спроектирована так, чтобы полагаться на репликацию по сети на другие узлы, а не на локальный диск. Попытка заставить её делать `fsync` превратит быстрый стриминг в медленную классическую СУБД.
    
- **Мониторинг размера ISR:** Настройте алерты на метрику `UnderReplicatedPartitions`. Если размер ISR падает ниже ожидаемого фактора репликации, ваша надежность находится под угрозой, а из-за специфики ISR замедление одного выжившего узла может затормозить продюсеров.
    

---

## 3. Реальные кейсы использования (Case Studies)

- **Сверхвысокая плотность партиций (Архитектура LinkedIn/Uber):** В огромных кластерах крутятся сотни тысяч партиций. Если бы Kafka использовала классический кворум большинства (как Raft/Paxos на уровне каждой партиции), метаданные и координация между узлами сожрали бы всю производительность. Модель ISR позволяет эффективно размазывать баланс лидерства по кластеру без тяжелого оверхеда.
    
- **Финансовые шины данных без потери пропускной способности:** Банковские системы используют связку `acks=all` + `ISR`, чтобы гарантировать Exactly-Once семантику. Поскольку подтверждение пишется в память (Page Cache) нескольких независимых серверов в разных стойках, система обеспечивает надежность уровня "Mainframe", сохраняя при этом миллионный RPS.
    

---

## 4. Рекомендации для продакшена (Production-Ready Recommendations)

- **Запретите грязные выборы:** Убедитесь, что параметр `unclean.leader.election.enable` выставлен в `false` (в современных версиях это дефолт). Если все реплики из ISR упадут, кластер лучше пусть временно откажет в доступности на запись, чем выберет лидером отставший узел и безвозвратно уничтожит или перемешает офсеты данных.
    
- **Тюнинг сетевой инфраструктуры брокеров:** Так как время коммита при `acks=all` ограничено самым медленным брокером в ISR, любой сетевой джиттер (network jitter) или "noisy neighbor" (шумный сосед по виртуализации) на фолловере мгновенно увеличит latency у продюсеров. Брокеры должны находиться в выделенной сети с минимальными задержками.
    
- **Изоляция дискового ввода-вывода (I/O):** Поскольку отвалившийся фолловер при возвращении в ISR начинает читать исторический лог с лидера большими кусками, этот процесс может вызвать всплеск утилизации дисков. Выносите логи Kafka на выделенные NVMe/SSD накопители, изолированные от системных логов ОС.



#### Unclean leader election and partition loss

Анализ раздела **«Unclean leader election and partition loss» (Грязные выборы лидера и потеря партиций)** из официальной документации [Kafka Replication](https://docs.confluent.io/kafka/design/replication.html#unclean-leader-election-and-partition-loss) компании Confluent.

---

### 1. Объяснение концепций (Explanation)

В этом фрагменте текста описывается критический сценарий отказа распределенной системы и фундаментальный компромисс между **доступностью (Availability)** и **согласованностью (Consistency)** согласно теореме CAP:

- **Нарушение гарантий:** Гарантия Kafka о невозможности потери данных действует только до тех пор, пока жива хотя бы одна синхронная реплика (ISR). Если одновременно или последовательно «умирают» абсолютно все узлы, хранящие копии конкретной партиции, эта гарантия аннулируется.
    
- **Дилемма полного отказа:** Когда все реплики уничтожены, у системы есть два полярных варианта поведения при восстановлении:
    
    1. **Ожидание «чистой» реплики:** Ждать, пока вернется в строй узел, который входил в состав ISR на момент аварии.
        
        - _Плюс:_ Сохраняется согласованность, подтвержденные сообщения не теряются.
            
        - _Минус:_ Партиция будет полностью недоступна для записи и чтения всё то время, пока сервер лежит. Если сервер уничтожен физически — данные заблокированы навсегда.
            
    2. **Грязные выборы (Unclean Leader Election):** Сделать лидером первую попавшуюся ожившую реплику, даже если она не входила в ISR и сильно отставала от лидера до аварии.
        
        - _Плюс:_ Высокая доступность. Работа восстанавливается максимально быстро.
            
        - _Минус:_ Лог этого отставшего узла становится новой «правдой». Все сообщения, которые он не успел скопировать до аварии, стираются навсегда (происходит потеря данных), а у потребителей ломаются офсеты.
            
- **Поведение по умолчанию:** Начиная с версии `0.11.0.0`, Kafka по умолчанию выбирает **первую стратегию** (консистентность в приоритете). Это поведение контролируется параметром `unclean.leader.election.enable`.
    

---

### 2. Лучшие практики (Best Practices)

- **Классификация топиков по критичности:** Никогда не применяйте одну стратегию ко всему кластеру «вслепую». Разделяйте конфигурации: для финансовых транзакций жестко выключайте грязные выборы, для логов и метрик — включайте.
    
- **Изоляция на уровне инфраструктуры:** Чтобы минимизировать саму вероятность сценария, когда упадут _все_ реплики, используйте механизм **Rack Awareness** (`broker.rack`). Распределяйте реплики одной партиции по разным физическим стойкам или зонах доступности (Availability Zones).
    
- **Контроль ретраев на продюсерах:** Если вы выбрали консистентность (`unclean.leader.election.enable=false`), продюсеры должны иметь адекватные настройки повторных попыток (`retries`), чтобы переждать кратковременные падения кластера, пока поднимается ISR-реплика.
    

---

### 3. Кейсы использования (Case Studies)

- **Кейс 1: Банковский сектор и Биллинг (Выбор Консистентности)**
    
    - _Сценарий:_ В результате аварии в главном дата-центре упали все три брокера, содержащие партицию топика `payment-orders`.
        
    - _Реализация:_ Настройка `unclean.leader.election.enable=false`. Система полностью останавливает обработку платежей по этой партиции. Администраторы и инженеры восстанавливают оригинальные серверы с их дисками.
        
    - _Результат:_ Бизнес несет операционные убытки из-за простоя, но балансы клиентов сходятся до копейки, так как ни один подтвержденный перевод не был потерян или перезаписан.
        
- **Кейс 2: IoT-Телеметрия и Мониторинг фабрики (Выбор Доступности)**
    
    - _Сценарий:_ На заводе произошел скачок напряжения, из-за чего перезагрузилась стойка с брокерами Kafka, обрабатывающими показатели датчиков температуры оборудования в реальном времени.
        
    - _Реализация:_ Настройка `unclean.leader.election.enable=true`. Один из брокеров поднялся быстрее остальных (хотя он немного отставал по логам). Он мгновенно стал лидером.
        
    - _Результат:_ Операторы на экранах продолжают видеть актуальную температуру станков прямо сейчас. Да, данные за последние 30 секунд перед аварией потерялись, но для real-time мониторинга критически важен текущий поток информации, а не история.
        

---

### 4. Рекомендации для продакшена (Production-Ready Recommendations)

- **Глобальный запрет на уровне кластера:** В файле `server.properties` всех брокеров всегда держите безопасный дефолт:
    
    Ini, TOML
    
    ```
    unclean.leader.election.enable=false
    ```
    
    Переопределяйте этот параметр в `true` только точечно для конкретных топиков через CLI-утилиту `kafka-configs.sh` по согласованию с владельцами бизнес-логики.
    
- **«Золотой стандарт» репликации:** Чтобы предотвратить сценарий полной смерти реплик, удерживайте конфигурацию: `replication.factor=3` и `min.insync.replicas=2`. Продюсеры при этом обязаны отправлять сообщения с `acks=all`.
    
- **Мониторинг критической метрики:** Настройте алерты (в Prometheus / Grafana) на метрику `UnderReplicatedPartitions`. Если число недореплицированных партиций растет — кластер находится в шаге от аварии, описанной в статье. Вы должны реагировать до того, как упадет последний выживший узел.
    
- **План аварийного ручного переключения (Runbook):** Если у вас топик заблокирован из-за `unclean...=false`, а диски упавших ISR-серверов сгорели безвозвратно, кластер зависнет навсегда. Администраторы должны иметь четкую инструкцию, как вручную перевести топик в режим `unclean...=true` через админ-панель или CLI, чтобы принудительно оживить систему, осознавая и принимая потерю данных.


###  Replica management

Анализ раздела **[Replica management](https://docs.confluent.io/kafka/design/replication.html#replica-management) (Управление репликами)** из официальной документации Confluent Kafka.

---

### 1. Объяснение концепций (Explanation)

В данном разделе рассматривается, как Kafka координирует работу огромного количества партиций в масштабах всего кластера, и как оптимизирован процесс восстановления при сбоях:

- **Масштабируемость и балансировка:** Один кластер Kafka управляет тысячами партиций. Чтобы избежать ситуации, когда высоконагруженные топики перегружают отдельные сервера, Kafka распределяет партиции между брокерами циклически (в режиме _round-robin_). Точно так же распределяется и лидерство (_leadership_), чтобы каждый узел брал на себя пропорциональную долю нагрузки по чтению и записи.
    
- **Контроллер кластера (Controller):** Окно времени, в течение которого выбирается новый лидер для партиции, — это критический период недоступности системы. Для его минимизации один из брокеров выбирается в качестве **Контроллера**.
    
- **Пакетные уведомления (Batching):** Именно Контроллер мгновенно обнаруживает падение любого брокера и централизованно меняет лидеров для всех затронутых партиций. За счет того, что уведомления о смене лидерства объединяются в пакеты (батчи), процесс выборов происходит практически мгновенно даже для тысяч партиций одновременно. При падении самого Контроллера его роль автоматически берет на себя один из выживших брокеров.
    

---

### 2. Лучшие практики (Best Practices)

- **Контроль распределения лидерства:** Всегда держите включенным параметр `auto.leader.rebalance.enable=true`. Это гарантирует, что после перезапуска упавшего брокера кластер автоматически вернет ему статус лидера для его «родных» партиций (Preferred Leaders), предотвращая перекос нагрузки.
    
- **Изоляция контроллеров (в архитектуре KRaft):** Для крупных продакшен-кластеров используйте выделенные узлы под роли контроллеров (кворум контроллеров в KRaft), на которых не хранятся пользовательские данные. Это защитит метаданные и процесс выборов лидеров от задержек, вызванных тяжелым дисковым или сетевым I/O на брокерах данных.
    
- **Равномерность партиционирования:** При проектировании топиков используйте сбалансированные ключи сообщений, чтобы данные распределялись по партициям равномерно. Даже идеальная балансировка партиций силами Kafka не поможет, если 90% трафика из-за плохого ключа пойдет в одну партицию на одном брокере.
    

---

### 3. Реальные кейсы использования (Case Studies)

- **Кейс 1: Крупные стриминговые платформы (борьба со «штормом выборов»):** В инфраструктурах уровня Netflix или Uber кластеры содержат десятки тысяч партиций. До оптимизации механизма Контроллера падение одного брокера приводило к лавинообразным индивидуальным выборам лидера для каждой партиции, что перегружало координационный слой и вызывало простои. Введение батчинга уведомлений на стороне Контроллера позволило переключать тысячи партиций за миллисекунды, делая сбои незаметными для клиентов.
    
- **Кейс 2: Динамические Multi-Tenant платформы:** В SaaS-системах, где под каждого клиента создаются свои топики, round-robin распределение партиций на уровне Контроллера предотвращает появление «горячих» брокеров. Новые клиенты распределяются по кластеру равномерно, исключая деградацию производительности «старых» изолированных узлов.
    

---

### 4. Рекомендации для продакшена (Production-Ready Recommendations)

- **Полный переход на KRaft (Kafka Raft):** Откажитесь от старой архитектуры на базе ZooKeeper. В режиме KRaft функции Контроллера интегрированы непосредственно в структуру метаданных Kafka, что сводит время восстановления и перевыборов лидеров к минимуму (до нескольких миллисекунд вместо секунд в ZooKeeper) и позволяет кластеру стабильно масштабироваться до миллионов партиций.
    
- **Мониторинг метрик Контроллера:** Настройте критические алерты в вашей системе мониторинга (Prometheus/Grafana) на следующие показатели:
    
    - `ActiveControllerCount` — строго должен быть равен `1`. Если равен `0` — кластер обезглавлен; если больше `1` — произошел опасный раскол сознания (split-brain).
        
    - `OfflinePartitionsCount` — в идеале равен `0`. Показывает количество партиций, оставшихся без лидера.
        
    - `LeaderElectionRateAndTimeMs` — время, затрачиваемое на перевыборы лидеров. Позволяет вовремя заметить деградацию производительности Контроллера.
        
- **Настройка лимитов дисбаланса:** Оставьте значение `leader.imbalance.per.broker.percentage=10` (дефолтное), но отрегулируйте частоту проверок `leader.imbalance.check.interval.seconds` (например, раз в 5–10 минут), чтобы автоматическая балансировка предпочтительных лидеров происходила плавно, без создания лишнего сетевого шума во время планового обслуживания инфраструктуры.
