**0:08** · There are three main pieces to a Kafka based architecture, the producer or where messages come from, the Kafka cluster itself and the consumers.

**0:52** · This ensures that messages are sent to the same partition and kept in the order that they were received.

**0:57** · The key is also used to serialize the message before being sent to Kafka and deserialized in the consumer.

**1:05** · Consumers subscribe to your topics and pull for messages.

**1:08** · One of the things that sets Kafka apart from other messaging systems is that once a message has been consumed, it is not deleted.

**1:15** · Instead, each message stays around until the retention period is met.

**1:20** · This is one of the main reasons why Kafka scales so easily.

**1:26** · One of consumer's most important and often misunderstood aspects is understanding offsets.

**1:32** · An offset is a unique identifier of a message in a partition.

**1:37** · It's an increasing number that's assigned to each message when it is stored.

**1:41** · Consumers can save which messages have been processed in a special Kafka topic that allows them to pick up where they left off in case something goes wrong.

### Consumer Group

**2:15** · Each consumer is a part of a consumer group.

**2:17** · Each group can be as small as one consumer or as large as the number of partitions in your Kafka topic.

**2:24** · Each consumer in a group adds parallelization for that task, splitting the messages among the consumers.

**2:30** · For example, let's say you have a topic that has a thousand messages arriving per second.

**2:35** · If your consumer can only process 500 messages per second, you would never catch up and you'd eventually fall significantly behind.

**2:43** · However, if you added another consumer to that group, you'd be able to handle all 1000 messages per second.

### Partition Assignment

**2:49** · Kafka automatically assigns consumers to partitions within a consumer group.

**2:54** · So if you had one partition, you'd be able to have one consumer.

**2:58** · If you had three partitions, you'd be able to have three consumers.

**3:02** · For instances where you have less than the maximum number of consumers, Kafka will assign consumers equally.

**3:08** · So if you had five partitions and only four consumers, each partition would be assigned to a consumer, with one consumer being assigned to two partitions.

**3:17** · This is the same thing that happens when you have a consumer that fails or is added.

**3:21** · A rebalancing takes place and assigns partitions to consumers.

**3:25** · If you need additional consumers in a group, you will also need to add more partitions to your topic.

**3:32** · It is also common to have a consumer act as a producer.

**3:35** · In this scenario you would have the application, consume the messages, perform any analysis, aggregation or other transformation, and then send it back to the separate Kafka topic.
