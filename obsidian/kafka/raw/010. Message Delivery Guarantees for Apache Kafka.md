## Kafka Message Delivery Guarantees

Apache Kafka® is an open-source distributed streaming system used for stream processing, real-time data pipelines, and data integration at scale.

This topic describes the semantic guarantees Kafka provides for message delivery between producers, brokers, and consumers. It also describes how Kafka supports exactly once delivery semantics, which is a key feature of Kafka that enables you to build reliable, fault-tolerant applications.

## Semantic guarantees

[Kafka Producer Design](https://docs.confluent.io/kafka/design/producer-design.html#producer-design) and [Kafka Consumer Design: Consumers, Consumer Groups, and Offsets](https://docs.confluent.io/kafka/design/consumer-design.html#consumer-design) helped you understand how producers and consumers work, but you should also understand the semantic guarantees Apache Kafka® provides between the broker and producers and consumers. Semantic guarantee refers to how the broker, producer and consumer agree to share messages. There are three types of message sharing:

- At most once: Messages are delivered once, and if there is a system failure, messages may be lost and are not redelivered.
- At least once: This means messages are delivered one or more times. If there is a system failure, messages are never lost, but they may be delivered more than once.
- Exactly once: This is the preferred behavior in that each message is delivered once and only once. Messages are never lost or read twice even if some part of the system fails.

These semantics offer tradeoffs between latency and message durability. You will need to choose the semantic that makes sense for your application context.

Also, note that many systems claim to provide exactly once delivery semantics, but this might not always what you think it is. For example, these system may not account for cases where a consumer or producer outside of the system has failed.

In contrast, Kafka’s semantics are straight-forward. When publishing a message Kafka considers the message *committed* to the log. After a message is committed it will not be lost as long as least one broker that has a replication of the partition with that message remains “alive”.

The next sections describe producer and consumer delivery in terms of these delivery semantics. These sections assume the broker is always lossless.

## Producer delivery

- **At most once** - (acks = 0) For the lowest latency, messages can be sent asynchronously in a “fire and forget” way, meaning the producer does not wait for any acknowledgement that messages were received, or for more latency, but less risk of message loss, the producer can wait for acknowledgment from the leader broker. In either case, messages are delivered once, and if there is a system failure, messages may be lost and are not redelivered.
- **At least once** - (acks = all)In this semantic, if a producer expects but fails to receive a response indicating that a message was committed, it will resend the message. This provides at-least-once delivery semantics since the message may be written to the log again during resending if the original request had succeeded. Since version 0.11.0.0, the Kafka producer provides an idempotent option for configuring message delivery. The idempotent delivery option guarantees that resending a message will not result in duplicate entries in the log, and that log order is maintained. To achieve this, the broker assigns each producer an ID and deduplicates messages using a sequence number that is sent by the producer with every message.
- **Exactly once** - (acks = -1 (all)) Starting with version 0.11.0.0, producers can utilize transactional delivery. This means a producer can request acknowledgment that messages were received and successfully replicated, and if it resends a message, it resends with idempotency, meaning existing messages are overwritten rather than duplicated. This comes with higher latency, but the most durability. In order to enable transactional guarantees, consumers should also be configured appropriately.

## Consumer receipt

Next, you should understand message semantics from the point-of-view of the consumer. Each message in a topic partition has a sequential identifier called an offset. All replicas of a partition have the same log with the same offsets, and the consumer controls its position in this log. If the consumer fails and another consumer needs to take over reading from this partition, the new consumer needs to know what offset it should start at. The consumer has a few options for how it reads messages and updates its current position:

- **At most once** - In this case, a consumer reads a set of messages, saves its position in the log, and then processes the messages. If the consumer process crashes after saving its position, but before saving the output of its message processing, the consumer that takes over processing would start at the saved position and the messages prior to that position would not be processed. This is *at-most-once* semantics for the consumer, because in a failure, messages may not be processed.
- **At least once** - This means a consumer reads a set of messages, processes the messages, and then saves its position. In this case, if the consumer process crashes after processing messages, but before saving its position, the new process that takes over may process some messages a second time. Note that in this case, you can assign messages a primary key, so that updates are idempotent, meaning that if the message is received twice, it just overwrites the existing record with the same message data.
- **Exactly once** - When consuming from a Kafka topic and producing to another topic such as in a Kafka Streams application, Kafka leverages transactional producer capabilities added in version 0.11.0.0 to achieve exactly once semantics. The consumer’s position is stored as a message in a topic, so offset data in written to Kafka in the same transaction as when processed data is written to the output topics. If the transaction is aborted, the consumer’s position reverts to its old value and you can specify whether output topics are visible to other consumers using the `isolation_level` property. For the default isolation level, `read_uncommitted`, all messages, even ones in aborted transactions, are visible to consumers. In `read_committed`, the consumer only reads messages from committed transactions.
	When writing to an external system, it can be challenging to coordinate the data the consumer is receiving and the consumer’s position in the data. Typically, this might be done with a two-phase commit, storing the consumer position, and then storing the consumed data. In contrast, Kafka stores the position in the same place it stores the consumed data. For example, a Connect connector populates data in Hadoop Distributed File System (HDFS) along with the offsets of the data it reads so that it is guaranteed that both data and offsets are updated, or neither is.

## Exactly once support

Kafka supports exactly-once delivery in [Kafka Streams](https://docs.confluent.io/platform/current/streams/overview.html) and uses transactional producers and consumers to provide exactly-once delivery when transferring and processing data between Kafka topics.

To enable exactly-once delivery for other consumer and producer systems, you can use the automatic offset management that [Kafka Connect API](https://docs.confluent.io/platform/current/connect/javadocs/javadoc/index.html) offers.

Otherwise, by default Kafka guarantees at-least-once delivery. You can implement at-most-once delivery by disabling retries on the producer and committing offsets in the consumer before processing a batch of messages.

## Перевод и объяснение раздела "Producer delivery"

В зависимости от настроек продюсера, Kafka предоставляет три режима доставки:

### 1. At most once (Максимум один раз)

- **Как это работает:** Для достижения минимальной задержки сообщения могут отправляться асинхронно в режиме «выстрелил и забыл» (fire and forget). Продюсер либо вообще не ждет подтверждения от брокера, либо ждет ответ только от брокера-лидера, но при этом у него отключены повторные попытки (`retries=0`).
    
- **Результат:** Сообщение отправляется ровно один раз. Если в сети произошел сбой или брокер упал в момент записи — данные будут безвозвратно утеряны, но повторно они никогда не отправятся.
    

### 2. At least once (Минимум один раз) — _Режим по умолчанию_

- **Как это работает:** Если продюсер отправил сообщение, но не получил от брокера подтверждения о том, что данные успешно записаны в лог (коммит), он отправляет этот пакет заново (`retries > 0`).
    
- **Проблема дубликатов:** Если брокер на самом деле успешно записал данные, но сетевой сбой произошел в момент отправки подтверждения (ack) назад к продюсеру, продюсер отправит данные повторно. На брокере появится дубликат.
    
- **Решение (Идемпотентность):** Начиная с версии Kafka 0.11.0.0, продюсер поддерживает опцию **идемпотентной доставки**. Она гарантирует, что переотправка батча из-за сетевых сбоев **не приведет к дубликатам в логе брокера**, а строгий порядок сообщений сохранится. Под капотом брокер присваивает каждому продюсеру уникальный ID (PID) и отслеживает порядковые номера сообщений (Sequence Numbers), отсекая дубли.
    

### 3. Exactly once (Строго один раз)

- **Как это работает:** Начиная с версии 0.11.0.0, продюсеры могут использовать **транзакционную доставку**. Продюсер отправляет сообщения внутри транзакции. Данные записываются с использованием механизма идемпотентности, а брокер гарантирует, что они будут подтверждены и реплицированы как единое целое.
    
- **Результат:** Сообщения либо запишутся строго один раз (без потерь и дубликатов), либо транзакция откатится, и данные не попадут к конечным потребителям (если они настроены правильно). Это самый надежный режим, но он платит за это повышенной задержкой (latency).

## Продакшн-рекомендации и Best Practices

Чтобы архитектура работала стабильно под нагрузкой, используйте следующие конфигурации в зависимости от критичности ваших данных.

### Сценарий А: Критически важные данные (Финансы, заказы, биллинг)

Вам категорически нельзя терять данные и нельзя допускать их дублирования.

- **`acks=all` (или `-1`)**: Продюсер будет ждать подтверждения не только от лидера, но и от всех синхронных реплик (`min.insync.replicas`). Это гарантирует защиту от потери данных при падении брокера.
    
- **`enable.idempotence=true`**: В современных версиях Kafka (начиная с 3.0+) этот параметр включен по умолчанию. Никогда не отключайте его для важных данных. Он защищает от дубликатов при ретраях сети.
    
- **`transactional.id`**: Задайте уникальный идентификатор транзакции для продюсера, если вы реализуете паттерн _«прочитал из одного топика -> обработал -> записал в другой топик»_ (в связке с Kafka Streams).
    

### Сценарий Б: Общие бизнес-метрики и логи (Высокая пропускная способность)

Потеря 0.001% данных при аварии кластера не критична, но дубликаты нежелательны, а скорость системы в приоритете.

- **`acks=all`** + **`enable.idempotence=true`**: Даже для обычных логов сегодня рекомендуется оставлять включенную идемпотентность, так как она практически не снижает производительность благодаря эффективному батчингу, но полностью избавляет от «мусора» в аналитических базах данных.
    
- Увеличьте параметры батчинга (`linger.ms=20` и `batch.size=65536`), чтобы компенсировать задержку от ожидания подтверждений брокера.
    

### Сценарий В: Телеметрия IoT, Real-time GPS (Минимальная задержка)

Данные идут огромным непрерывным потоком. Если один пакет потерялся — ничего страшного, так как через секунду придет новый, более актуальный.

- **`acks=0`**: Продюсер не ждет ответов. Максимальная скорость, нулевая гарантия доставки (чистый _At-most-once_).

## Кейс-стади (Case Studies) из реальной практики

### Кейс 1: Как распределенная транзакция спасает банковский перевод

**Задача:** Сервер обрабатывает событие «Списание со счета А» и должен отправить событие «Зачисление на счет Б» в другой топик Kafka. Если сервер упадет посередине, деньги спишутся, но не зачислятся.

**Решение:** Использование `Exactly-once` семантики. Продюсер открывает транзакцию. В рамках одной транзакции он отправляет сообщение о зачислении на счет Б и одновременно коммитит офсет прочитанного сообщения о списании. Если приложение падает, брокер Kafka откатывает всю транзакцию. Деньги не теряются, а после перезапуска процесс начинается заново без дубликатов.

### Кейс 2: Проблема «Сетевого шторма» при отключенной идемпотентности

**Задача:** Крупный ритейлер использовал старую версию Kafka с дефолтными настройками _At-least-once_ (без идемпотентности).

**Что пошло не так:** Во время распродажи дата-центр испытал кратковременные проблемы с сетью (микро-таймауты). Продюсеры отправляли батчи с заказами, брокер их записывал, но подтверждения терялись в сети. Продюсеры честно делали ретраи. В результате в базу данных аналитики заказы записались по 2–3 раза. Система посчитала, что товаров продано больше, чем было на складе.

**Решение:** Перевод продюсеров на `enable.idempotence=true` полностью решил проблему. При повторных сетевых штормах брокер просто отбрасывал дублирующиеся батчи, возвращая продюсеру успешный статус.



Твоя схема близка к истине, но в ней есть несколько критически важных нюансов, которые в продакшене могут привести к потере данных.

Давай опровергнем устаревшие мифы и разберем, как это работает в современной Kafka.

## Подтверждаю или опровергаю?

### 1. `At most once` -> `acks = 0`

**ПОДТВЕРЖДАЮ.**

Это чистый режим «выстрелил и забыл» (fire and forget). Продюсеру вообще не важен ответ брокера. Если пакет потерялся в сети или у брокера упал диск — данные ушли в никуда. Семантика «максимум один раз» соблюдена: либо 0 раз (потеря), либо 1 раз (успех), но дубликатов точно не будет.

### 2. `At least once` -> `acks = 1`

**ОПРОВЕРГАЮ (с уточнением).**

Раньше в старых гайдах так писали, но это **опасное заблуждение**.

- Если выставить `acks=1` и включить ретраи, то при сетевых сбоях продюсер действительно будет переотправлять данные, что дает _попытку_ достучаться.
    
- **Почему это НЕ гарантирует At-least-once:** Если брокер-лидер принял батч, записал себе на диск, отправил продюсеру подтверждение (ack) и _в эту же миллисекунду умер_ (до того, как реплики успели скопировать данные) — сообщение потеряно навсегда. Продюсер думает, что всё ок, и больше ничего не переотправит. Это превращается в _At-most-once_.
    
- **Как правильно:** В современной Kafka гарантия _At-least-once_ по умолчанию работает на **`acks=all`** (или `acks=-1`), но с отключенной идемпотентностью (хотя отключать её сейчас нет никакого смысла).
    

### 3. `Exactly once` -> `acks = all`

**ПОДТВЕРЖДАЮ, но это лишь 25% успеха.**

Настройка `acks=all` — это обязательный, фундаментальный шаг для Exactly-Once Semantics (EOS), но сама по себе она этот режим не включает. Чтобы получить «строго один раз», тебе нужен целый квартет настроек:

1. `acks=all` (чтобы данные гарантированно были у всех реплик).
    
2. `enable.idempotence=true` (чтобы брокер отсекал дубликаты при сетевых ретраях).
    
3. Использование транзакционного продюсера (`transactional.id` в конфиге и методы `beginTransaction()` / `commitTransaction()` в коде).
    
4. `isolation.level=read_committed` на стороне консьюмера (чтобы он не читал откаченные транзакции).

## С каким `acks` работает `min.insync.replicas`?

Параметр `min.insync.replicas` работает **СТРОГО И ТОЛЬКО с `acks=all`** (или `acks=-1`).

С другими режимами он просто игнорируется:

- Если `acks=0`, продюсер не ждет ответа вообще.
    
- Если `acks=1`, брокер-лидер проверяет только свою локальную запись и сразу отвечает. Ему всё равно, сколько там реплик «живы».
    

### Как они работают в тандеме при `acks=all`:

`min.insync.replicas` (настройка на уровне топика или брокера) — это **минимальный порог безопасности**.

Допустим, у тебя фактор репликации топика равен 3, а `min.insync.replicas=2`.

1. Продюсер шлет батч с `acks=all`.
    
2. Брокер-лидер смотрит на свой кластер и проверяет: _«Сколько реплик сейчас живы и успевают за мной (находятся в ISR)?»_
    
3. Если живы лидер и хотя бы одна реплика (итого 2), он записывает данные, ждет копирования на эту реплику и возвращает продюсеру успешный `ack`.
    
4. Если вдруг две реплики упали, и лидер остался один в списке ISR (а у нас в конфиге стоит минимум 2), лидер **откажется принимать данные** и вернет продюсеру ошибку `NotEnoughReplicasException`.


### Consumer receipt
Выделенный раздел **«Consumer receipt» (Получение данных потребителем)** из [документации Confluent](https://docs.confluent.io/kafka/design/delivery-semantics.html#exactly-once-support) объясняет семантику доставки сообщений с точки зрения принимающей стороны (Consumer). То, как и в какой момент приложение сохраняет свою позицию (офсет), определяет надежность всей системы.

Ниже приведен подробный разбор текста на русском языке, а также лучшие практики, кейсы и готовые рекомендации для продакшена.

## Перевод и объяснение раздела "Consumer receipt"

Каждое сообщение в партиции имеет порядковый номер — офсет. Консьюмер сам управляет своей позицией в этом логе. Если один консьюмер падает, новый инстанс должен знать, с какого офсета ему продолжить чтение. Текст выделяет три сценария:

### 1. At most once (Максимум один раз)

- **Алгоритм:** Получатель читает пачку сообщений $\rightarrow$ **сразу сохраняет (коммитит) офсет** в Kafka $\rightarrow$ начинает обрабатывать сообщения.
    
- **Проблема при сбое:** Если приложение упадет в процессе обработки данных, новый инстанс после перезапуска начнет работу со следующего (уже закоммиченного) офсета. Сообщения, которые не успели обработаться, **будут безвозвратно утеряны**.
    

### 2. At least once (Минимум один раз) — _Режим по умолчанию_

- **Алгоритм:** Получатель читает сообщения $\rightarrow$ **полностью их обрабатывает** $\rightarrow$ сохраняет офсет в Kafka.
    
- **Проблема при сбое:** Если приложение упадет _после_ обработки сообщений, но _до_ того, как успеет отправить коммит офсета на брокер, новый инстанс прочитает эти сообщения заново. Возникнут **дубликаты**.
    
- **Решение:** Текст рекомендует использовать первичные ключи (Primary Keys), чтобы сделать обработку идемпотентной (повторный ивент просто перезапишет старый, а не создаст дубликат).
    

### 3. Exactly once (Строго один раз)

- **Алгоритм:** Используется в сценариях «из Kafka в Kafka» (например, в Kafka Streams). Офсет консьюмера записывается в служебный топик в рамках **той же самой транзакции**, в которой продюсер пишет результаты обработки в целевой топик. Если транзакция откатывается, офсет возвращается назад.
    
- **Важность изоляции:** Чтобы другие консьюмеры не видели промежуточные данные, они должны работать в режиме `isolation.level = read_committed`.
    

### Взаимодействие с внешними системами

Когда данные нужно записать не в Kafka, а во внешнюю базу (PostgreSQL, Elasticsearch, HDFS), скоординировать сохранение офсета и запись данных сложно. Обычно используют двухфазный коммит (2PC). Но Kafka Connect делает умнее: он **сохраняет офсеты в саму целевую систему** в одном пакете с данными. Например, в HDFS данные и их офсеты пишутся атомарно: либо запишется и то и другое, либо ничего.

## Продакшн-рекомендации и Best Practices

### 1. Отключите Auto-Commit в продакшене

По умолчанию свойство `enable.auto.commit` равно `true` (Kafka сбрасывает офсеты каждые 5 секунд). Это чистый рандом: вы можете получить как утерю данных (_At-most-once_), так и тонны дубликатов (_At-least-once_).

- **Рекомендация:** Всегда выставляйте `enable.auto.commit = false`. Переходите на ручное управление коммитами (`commitSync` или `commitAsync`).
    

### 2. Стратегия ручного коммита (Батчинг)

Коммитить после каждого отдельного сообщения (`commitSync()`) — архитектурная смерть для производительности, так как приложение будет постоянно блокировать поток сетевыми запросами к брокеру.

- **Рекомендация:** Обрабатывайте данные пачками (в цикле по `ConsumerRecords`). Вызывайте `commitSync()` **строго в конце обработки всего батча**, полученного за один `poll()`. Это обеспечит контролируемую семантику _At-least-once_ с минимальными дубликатами (в пределах одной пачки).
    

### 3. Дедупликация на уровне Идемпотентного Приемника

Если вы пишете в базу данных (SQL/NoSQL), не полагайтесь на то, что Kafka никогда не пришлет дубликат при аварии.

- **Рекомендация:** Конструируйте детерминированные ключи документов. Для SQL используйте конструкцию `INSERT ... ON CONFLICT DO UPDATE` (Upsert). Для MongoDB/Elasticsearch используйте бизнес-ID сообщения в качестве `_id` документа.

## Кейс-стади (Case Studies) из реальной практики

### Кейс 1: Утеря финансовых транзакций из-за Auto-Commit

- **Что произошло:** Микросервис обработки платежей работал с дефолтным `enable.auto.commit = true`. Во время обработки тяжелого батча транзакций (занимавшей 7 секунд), на 5-й секунде Kafka автоматически закоммитила офсет. На 6-й секунде под к подам Kubernetes пропало питание. Pod упал.
    
- **Результат:** Новый поднявшийся под начал чтение с закоммиченного офсета. Часть платежей за эти 2 секунды осталась необработанной, деньги «зависли».
    
- **Решение:** Перевод на `enable.auto.commit = false` и вызов `commitSync()` только после подтверждения от банковского шлюза.
    

### Кейс 2: Синхронизация с базой данных через паттерн Transactional Outbox (Борьба с 2PC)

- **Задача:** Команда пыталась реализовать _Exactly-once_ при записи данных из Kafka в MySQL. Они обновляли баланс пользователя в MySQL, а затем коммитили офсет в Kafka. При падении сети между MySQL и Kafka баланс обновлялся, а коммит не долетал — возникал дубликат. Двухфазный коммит (2PC) сильно тормозил систему.
    
- **Результат:** Команда применила рекомендацию из конца статьи — объединить хранение. Они создали в MySQL служебную таблицу `kafka_offsets`. Теперь в рамках одной стандартной ACID-транзакции MySQL они обновляли баланс пользователя и записывали текущий `partition` и `offset` в таблицу `kafka_offsets`.
    
- **Итог:** При перезапуске консьюмер сначала читал из MySQL его «личный» закоммиченный офсет, делал `consumer.seek()` на эту позицию и продолжал работу. Дубликаты исчезли на 100% без использования тяжелых распределенных транзакций.
